# Data Synchronization Reference

This document explains how the system maintains temporal alignment across all data streams for post-hoc neural analysis.

---

## The Synchronization Problem

The experiment generates six concurrent data streams that must be aligned to sub-millisecond precision for neural analysis:

| Stream | Device | Native clock |
|--------|--------|-------------|
| Neural spikes / LFP | Neuropixel + recording system | Recording system clock |
| Face video | NI IMAQ cam0 | Camera hardware clock |
| Whisker video | NI IMAQ cam2 | Camera hardware clock |
| DAQmx digital events | NI Dev2 | NI DAQ clock |
| DAQmx audio output | NI Dev2 | NI DAQ clock |
| Behavior labels | Bonsai software | PC system clock |

All streams use different clocks with independent drift. The solution is to anchor every stream to the NI DAQmx clock via TTL event markers sampled at 150 kHz.

---

## Synchronization Architecture

```
                    ┌──────────────────────────────────┐
                    │       NI DAQmx (Dev2)            │
                    │  150 kHz digital reference clock  │
                    └─────────────┬────────────────────┘
                                  │
               ┌──────────────────┼──────────────────┐
               │                  │                   │
               ▼                  ▼                   ▼
    port0/line6 (Face cam)  port0/line5 (Whisk cam)  port0/line4 (Events)
        HIGH on each            HIGH on each           HIGH on each
        frame acquired          frame acquired         trial/stimulus event
               │                  │                   │
               ▼                  ▼                   ▼
    Recorded by neural       Recorded by neural    Recorded by neural
    recording system         recording system      recording system
    as digital channel       as digital channel    as digital channel
```

Every frame from both cameras generates a TTL pulse on a DAQmx line. The neural recording system (sampling at its own rate, typically 30 kHz) records these pulses alongside spikes, creating a frame-by-frame alignment bridge.

## Event Marker Channels

### Digital lines → neural recording (150 kHz TTL)

| Dev2 Line | Signal | Purpose |
|-----------|--------|---------|
| 0 | Record | HIGH = recording session active; timestamps session boundaries |
| 2 | At-Whiskers | HIGH when linear motor confirmed at whisker position |
| 3 | At-Object | HIGH when object wheel at target; marks stimulus onset |
| 4 | Event | Condition marker; HIGH for stimulus-type events |
| 5 | Whisker-Cam | Pulses HIGH on each whisker camera frame acquisition |
| 6 | Face-Cam | Pulses HIGH on each face camera frame acquisition |
| 7 | Whisking | HIGH when animal classified as whisking |

### Audio output alignment

The audio waveform is generated by the same DAQmx device (Dev2/ao0) that drives the digital sync lines. Both run from the same 250 kHz master clock, so audio onset is inherently aligned with digital event markers to within one sample (4 µs).

## Camera Frame Alignment

### Step 1: Frame index CSVs

Bonsai writes a CSV for each camera that records the **acquisition frame index** for every frame:

- `{ID}_Face.csv` — one row per face camera frame
- `{ID}_Whiskers.csv` — one row per whisker camera frame

### Step 2: TTL pulses on DAQmx lines 5 and 6

Each camera frame also triggers a HIGH pulse on the corresponding DAQmx digital line. The neural recording system records these as a digital channel at its sampling rate.

### Step 3: Post-hoc alignment

Given:
- `T_frame[n]` = timestamp of the n-th TTL pulse on the neural recording system's digital channel
- Frame index CSV maps frame number → PC acquisition time

The two can be matched by frame count: frame N in the video corresponds to the N-th rising edge on the DAQmx line recorded by the neural system. This allows any video frame to be placed on the neural recording timeline.

---

## Behavioral Event Alignment

### Trial events → `_conditions.csv`

Each trial logs:
- Condition index (0–4)
- PC system timestamp

These timestamps are in PC clock time (Windows performance counter). To align with neural data:

1. The "Record" DAQmx line (line 0) provides session start/stop boundaries recorded on the neural system timeline
2. Map PC timestamps to neural timeline assuming linear drift within a session (usually <1 ms/min for NI devices)
3. For precise alignment, use the "At-Object" TTL pulse (line 3) which fires at the same moment `_conditions.csv` is written

### Whisking state → DAQmx line 7

The whisking classifier output (`endW` BehaviorSubject) directly drives DAQmx line 7. This means the whisking state is recorded at 150 kHz resolution on the neural recording system, requiring no separate alignment step.

## Barcode Synchronization (COM9)

At experiment end (`F9`), Bonsai sends `q` and `s` to the barcode Arduino on COM9. This device emits a synchronization barcode that can be captured by:

- A photodiode connected to the neural recording system
- An analog or digital input on the neural recording system

The barcode encodes the PC system time of the end-of-experiment event, enabling a global anchor point for retrospective clock correction across the full session.

```python
conditions['neural_time'] = trial_neural_times[:len(conditions)]

# 5. Whisking state is already on neural timeline (line 7)
# Downsample or threshold as needed for analysis
```

## Clock Drift Estimation

For sessions > 30 minutes, clock drift between the PC and NI device should be measured. Procedure:

1. Note session start PC timestamp (from Bonsai log)
2. Note session start neural time (first rising edge of line 0 "Record")
3. Note session end PC timestamp
4. Note session end neural time (falling edge of line 0 "Record")
5. Linear interpolation between these two anchors corrects all intermediate PC timestamps

Typical drift: <10 ms per hour for NI USB devices under Windows.
